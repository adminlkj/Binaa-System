   149вЖТ## рЯФі CRITICAL ISSUES (7)
   150вЖТ
   151вЖТ### C1. Float Used for ALL Financial Fields (138 occurrences)
   152вЖТ- **Severity:** рЯФі Critical - Data integrity risk
   153вЖТ- **Models affected:** ALL 62 models with financial/monetary fields
   154вЖТ- **Count:** 138 Float fields used for money, rates, quantities
   155вЖТ- **Problem:** IEEE 754 Float has precision errors (0.1 + 0.2 вЙ† 0.3). In an ERP, SAR 1,000,000.15 can become 1000000.14999999. Over thousands of transactions, rounding errors accumulate.
   156вЖТ- **Examples:** JournalLine.debit/credit, SalesInvoice.totalAmount, Contract.value, Employee.basicSalary, Equipment.purchasePrice
   157вЖТ- **Fix:** Change all monetary Float fields to Decimal (Prisma emulates Decimal as TEXT in SQLite вАФ still more precise than Float). For rates/percentages, Decimal is also preferred.
   158вЖТ- **Note:** SQLite limitation вАФ Prisma stores Decimal as TEXT but performs correct arithmetic in JS.
   159вЖТ
   160вЖТ### C2. Zero Database Indexes (only 1 @@unique, no @@index)
   161вЖТ- **Severity:** рЯФі Critical - Performance & scalability
   162вЖТ- **Problem:** Only 1 composite @@unique found (PeriodClosing year+month+type). ZERO @@index declarations.
   163вЖТ- **Impact:** Every query filtering by foreign key (projectId, clientId, supplierId, equipmentId, etc.) performs a full table scan. With production data, this will cause severe slowdowns.
   164вЖТ- **Missing indexes on FK columns (35+):** Project.branchId, Project.clientId, Contract.projectId, SalesInvoice.clientId, SalesInvoice.projectId, PurchaseOrder.supplierId, PurchaseOrder.projectId, EquipmentRental.contractId, EquipmentRental.equipmentId, EquipmentRental.clientId, Timesheet.rentalId, Timesheet.contractId, JournalLine.journalEntryId, JournalLine.accountId, Expense.projectId, Employee.branchId, and 20+ more.
   165вЖТ- **Missing indexes on frequently queried fields:** status fields (all models), date fields, invoiceNo, entryNo, code fields.
   166вЖТ- **Fix:** Add @@index on all FK columns and frequently filtered/ordered columns.
   167вЖТ
   168вЖТ### C3. Broken Referential Integrity вАФ SubcontractorContract.projectId Has No @relation
   169вЖТ- **Severity:** рЯФі Critical - Data integrity
   170вЖТ- **Model:** SubcontractorContract has `projectId String` but NO `@relation` defined, and Project has no `subcontractorContracts` back-relation.
   171вЖТ- **Impact:** No referential integrity enforcement, no Prisma include/join queries possible, orphan records possible.
   172вЖТ- **Fix:** Add `project Project @relation(fields: [projectId], references: [id])` and `subcontractorContracts SubcontractorContract[]` on Project.
   173вЖТ
   174вЖТ### C4. Missing onDelete/onUpdate on Most Relations (only 7 of 60+)
   175вЖТ- **Severity:** рЯФі Critical - Data integrity
   176вЖТ- **Only 7 relations have onDelete: Cascade:** TeamMemberвЖТWorkTeam, SalesInvoiceItemвЖТSalesInvoice, PurchaseRequestItemвЖТPurchaseRequest, PurchaseOrderItemвЖТPurchaseOrder, GoodsReceiptItemвЖТGoodsReceipt, PurchaseInvoiceItemвЖТPurchaseInvoice, JournalLineвЖТJournalEntry.
   177вЖТ- **Missing critical cascades:**
   178вЖТ  - Deleting a Client should Restrict if invoices/payments exist
   179вЖТ  - Deleting a Project should Restrict if contracts/invoices exist
   180вЖТ  - Deleting an Equipment should Restrict if active rentals exist
   181вЖТ  - Deleting a Contract should Cascade to EquipmentRental
   182вЖТ- **Zero onUpdate rules** defined anywhere.
   183вЖТ- **Fix:** Add explicit onDelete: Restrict on all financial parent entities, onDelete: Cascade on line-item children, onUpdate: Cascade on all.
   184вЖТ
   185вЖТ### C5. No Soft Delete (deletedAt) on Any Model
   186вЖТ- **Severity:** рЯФі Critical - Data loss risk
   187вЖТ- **Problem:** All deletes are hard deletes. In ERP systems, financial records (invoices, journal entries, payments) should NEVER be hard-deleted.
   188вЖТ- **Impact:** Accidental deletion of financial data is unrecoverable. Audit trail is broken.
   189вЖТ- **Fix:** Add `deletedAt DateTime?` to all financial models and implement soft-delete filtering at the application layer or via Prisma middleware.
   190вЖТ
   191вЖТ### C6. FixedAsset Account References Have No @relation to Account
   192вЖТ- **Severity:** рЯФі Critical - Data integrity
   193вЖТ- **Model:** FixedAsset has `accountId String?`, `depExpenseAccountId String?`, `accumDepAccountId String?` вАФ all without @relation to Account.
   194вЖТ- **Impact:** No referential integrity, no Prisma join queries, possible orphan references if an Account is deleted.
