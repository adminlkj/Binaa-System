   149→## 🔴 CRITICAL ISSUES (7)
   150→
   151→### C1. Float Used for ALL Financial Fields (138 occurrences)
   152→- **Severity:** 🔴 Critical - Data integrity risk
   153→- **Models affected:** ALL 62 models with financial/monetary fields
   154→- **Count:** 138 Float fields used for money, rates, quantities
   155→- **Problem:** IEEE 754 Float has precision errors (0.1 + 0.2 ≠ 0.3). In an ERP, SAR 1,000,000.15 can become 1000000.14999999. Over thousands of transactions, rounding errors accumulate.
   156→- **Examples:** JournalLine.debit/credit, SalesInvoice.totalAmount, Contract.value, Employee.basicSalary, Equipment.purchasePrice
   157→- **Fix:** Change all monetary Float fields to Decimal (Prisma emulates Decimal as TEXT in SQLite — still more precise than Float). For rates/percentages, Decimal is also preferred.
   158→- **Note:** SQLite limitation — Prisma stores Decimal as TEXT but performs correct arithmetic in JS.
   159→
   160→### C2. Zero Database Indexes (only 1 @@unique, no @@index)
   161→- **Severity:** 🔴 Critical - Performance & scalability
   162→- **Problem:** Only 1 composite @@unique found (PeriodClosing year+month+type). ZERO @@index declarations.
   163→- **Impact:** Every query filtering by foreign key (projectId, clientId, supplierId, equipmentId, etc.) performs a full table scan. With production data, this will cause severe slowdowns.
   164→- **Missing indexes on FK columns (35+):** Project.branchId, Project.clientId, Contract.projectId, SalesInvoice.clientId, SalesInvoice.projectId, PurchaseOrder.supplierId, PurchaseOrder.projectId, EquipmentRental.contractId, EquipmentRental.equipmentId, EquipmentRental.clientId, Timesheet.rentalId, Timesheet.contractId, JournalLine.journalEntryId, JournalLine.accountId, Expense.projectId, Employee.branchId, and 20+ more.
   165→- **Missing indexes on frequently queried fields:** status fields (all models), date fields, invoiceNo, entryNo, code fields.
   166→- **Fix:** Add @@index on all FK columns and frequently filtered/ordered columns.
   167→
   168→### C3. Broken Referential Integrity — SubcontractorContract.projectId Has No @relation
   169→- **Severity:** 🔴 Critical - Data integrity
   170→- **Model:** SubcontractorContract has `projectId String` but NO `@relation` defined, and Project has no `subcontractorContracts` back-relation.
   171→- **Impact:** No referential integrity enforcement, no Prisma include/join queries possible, orphan records possible.
   172→- **Fix:** Add `project Project @relation(fields: [projectId], references: [id])` and `subcontractorContracts SubcontractorContract[]` on Project.
   173→
   174→### C4. Missing onDelete/onUpdate on Most Relations (only 7 of 60+)
   175→- **Severity:** 🔴 Critical - Data integrity
   176→- **Only 7 relations have onDelete: Cascade:** TeamMember→WorkTeam, SalesInvoiceItem→SalesInvoice, PurchaseRequestItem→PurchaseRequest, PurchaseOrderItem→PurchaseOrder, GoodsReceiptItem→GoodsReceipt, PurchaseInvoiceItem→PurchaseInvoice, JournalLine→JournalEntry.
   177→- **Missing critical cascades:**
   178→  - Deleting a Client should Restrict if invoices/payments exist
   179→  - Deleting a Project should Restrict if contracts/invoices exist
   180→  - Deleting an Equipment should Restrict if active rentals exist
   181→  - Deleting a Contract should Cascade to EquipmentRental
   182→- **Zero onUpdate rules** defined anywhere.
   183→- **Fix:** Add explicit onDelete: Restrict on all financial parent entities, onDelete: Cascade on line-item children, onUpdate: Cascade on all.
   184→
   185→### C5. No Soft Delete (deletedAt) on Any Model
   186→- **Severity:** 🔴 Critical - Data loss risk
   187→- **Problem:** All deletes are hard deletes. In ERP systems, financial records (invoices, journal entries, payments) should NEVER be hard-deleted.
   188→- **Impact:** Accidental deletion of financial data is unrecoverable. Audit trail is broken.
   189→- **Fix:** Add `deletedAt DateTime?` to all financial models and implement soft-delete filtering at the application layer or via Prisma middleware.
   190→
   191→### C6. FixedAsset Account References Have No @relation to Account
   192→- **Severity:** 🔴 Critical - Data integrity
   193→- **Model:** FixedAsset has `accountId String?`, `depExpenseAccountId String?`, `accumDepAccountId String?` — all without @relation to Account.
   194→- **Impact:** No referential integrity, no Prisma join queries, possible orphan references if an Account is deleted.
   195→- **Fix:** Add named relations: `assetAccount Account? @relation("FixedAssetAccount", ...)`, `depExpenseAccount Account? @relation("FixedAssetDepExpense", ...)`, `accumDepAccount Account? @relation("FixedAssetAccumDep", ...)`.
   196→
   197→### C7. BankAccount.accountId Has No @relation to Account
   198→- **Severity:** 🔴 Critical - Data integrity
   199→- **Model:** BankAccount has `accountId String?` with comment "Link to Chart of Accounts" but no @relation.
   200→- **Impact:** Same as C6 — no referential integrity, no Prisma joins.
   201→- **Fix:** Add `account Account? @relation("BankAccountGL", fields: [accountId], references: [id])` and back-relation on Account.

   212→### M2. String Instead of Enum for 20+ Status/Type Fields
   213→- **Models affected:** EquipmentRental (status, pricingType, operationMode, fuelResponsibility, insuranceResponsibility, deliveryFeesType), Salary.activityType, Expense.activityType, PurchaseInvoice.activityType, FixedAsset (status, category, depreciationMethod), Provision (type, status), ProvisionMovement.movementType, BankTransaction.transactionType, BankReconciliation.status, PeriodClosing (type, status), Expense.payFrom, ClientPayment.receivedIn, SupplierPayment.paidFrom
   214→- **Impact:** No type safety, invalid values possible, harder to query/refactor.
   215→- **Fix:** Define proper enums for each status/type field.
   216→
   217→### M3. Missing User/Authentication Model
   218→- **Problem:** No User, Role, or Permission model. AuditLog has `userId String?` with no relation.
   219→- **Impact:** No authentication/authorization, no way to track who performed actions, no role-based access control.
   220→- **Fix:** Add User model with role-based access, and add `createdBy String?` to key models.
   221→
   222→### M4. Missing Payment-Invoice Many-to-Many Link
   223→- **Problem:** ClientPayment has one `invoiceId`, SupplierPayment has one `invoiceId`. In practice, a single payment can cover multiple invoices, and one invoice can have multiple partial payments.
   224→- **Impact:** Cannot properly allocate payments across invoices. Payment-on-account not supported.
   225→- **Fix:** Create PaymentAllocation junction model (paymentId, invoiceId, allocatedAmount).
   226→
   227→### M5. SupplierPayment Missing Relation to PurchaseInvoice
   228→- **Problem:** SupplierPayment has `invoiceId String?` but no @relation to PurchaseInvoice, and PurchaseInvoice has no back-relation to SupplierPayment.
   229→- **Impact:** Cannot track which supplier payments apply to which purchase invoices.
   230→- **Fix:** Add proper relation (or PaymentAllocation junction as in M4).
   231→
   232→### M6. SubcontractorInvoice Not Linked to SubcontractorContract
   233→- **Problem:** SubcontractorInvoice links to Subcontractor and Project, but not to SubcontractorContract.
   234→- **Impact:** Cannot track which invoices belong to which subcontractor contract.
   235→- **Fix:** Add `contractId String` with @relation to SubcontractorContract.
   236→
   237→### M7. JournalEntry.reversedEntryId Has No Self-Relation
   238→- **Problem:** JournalEntry has `reversedEntryId String?` but no self-relation defined.
   239→- **Impact:** Cannot use Prisma include to fetch the reversed entry.
   240→- **Fix:** Add `reversedEntry JournalEntry? @relation("JournalReversal", fields: [reversedEntryId], references: [id])` and `reversals JournalEntry[] @relation("JournalReversal")`.
   241→
   242→### M8. Account Model Missing Critical Fields
   243→- **Problem:** Account model lacks `openingBalance`, `currentBalance` (cached), and `updatedAt`.
   244→- **Impact:** Cannot store opening balances per fiscal year. Balance must always be computed on the fly (slow for large datasets).
   245→- **Fix:** Add `openingBalance Decimal @default(0)`, `updatedAt DateTime @updatedAt`, and consider a separate AccountBalance model per fiscal period.
   246→
   247→### M9. No Inventory Transaction/Movement Model
   248→- **Problem:** InventoryItem has `quantity Float` but no StockMovement or InventoryTransaction model to track in/out history.
   249→- **Impact:** Cannot trace inventory movements, no FIFO/LIFO/AVG cost tracking, no stock audit trail.
   250→- **Fix:** Add InventoryTransaction model (itemId, type IN/OUT, quantity, unitCost, referenceType, referenceId, date).
   251→
   252→### M10. No Fiscal Year / Fiscal Period Model
   253→- **Problem:** PeriodClosing exists but is minimal. No proper FiscalYear/FiscalPeriod model with opening/closing balances.
   254→- **Impact:** Cannot properly handle year-end closing, opening balances, or multi-year accounting.
   255→- **Fix:** Add FiscalYear model (year, startDate, endDate, status) and FiscalPeriod model (fiscalYearId, periodNumber, startDate, endDate, status).

   547→## 🔴 CRITICAL ISSUES (8)
   548→
   549→### 1. No Database Transactions Across All Write Endpoints
   550→Almost every POST/PUT/DELETE performs multiple DB operations (create record + create journal entry + update related records) without Prisma `$transaction`. Examples:
   551→- `client-payments POST`: payment create + journal entry + invoice update (3 separate operations)
   552→- `sales-invoices POST`: invoice create + items create + journal entry + timesheet update (4+ operations)
   553→- `supplier-payments POST`: payment create + journal entry + invoice update
   554→- Any failure mid-operation leaves data inconsistent (e.g., invoice marked INVOICED but no journal entry exists)
   555→
   556→### 2. Silent Accounting Entry Failures
   557→All financial endpoints (invoices, payments, expenses, progress-claims) catch journal entry creation errors and continue:
   558→```typescript
   559→} catch (accountingError) {
   560→  console.error('Accounting entry failed...', accountingError)
   561→  // Don't fail the main operation — INVOICE CREATED WITHOUT JOURNAL ENTRY
   562→}
   563→```
   564→This breaks the fundamental accounting trail. Records can exist without corresponding journal entries, making financial reports inaccurate.
   565→
   566→### 3. VAT State Machine Inconsistency
   567→Two conflicting state machines exist:
   568→- `/api/vat/route.ts` PATCH: DRAFT → FILED → PAID
   569→- `/api/vat/[id]/route.ts` PATCH: DRAFT → CREATED → DUE → PAID
   570→The front-end may call either endpoint, leading to unpredictable state transitions.
   571→
   572→### 4. N+1 Query Problem in Equipment Timesheets GET
   573→`/api/equipment/timesheets/route.ts` GET makes a separate DB query for EACH timesheet to fetch client name:

   577→    const rentalWithClient = await db.equipmentRental.findUnique(...)
   578→    const cl = await db.client.findUnique(...)
   579→    ...
   580→  })
   581→)
   582→```

   703→```typescript
